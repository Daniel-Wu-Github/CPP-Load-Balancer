System Architecture and Workflow Description

The system architecture is centered around the LoadBalancer class which acts as the primary controller for the simulation. This class manages a queue of Request objects and maintains a dynamic vector of WebServer instances. Each Request object encapsulates data including source and destination IP addresses, the required processing time, and the job type. The WebServer class represents an individual processing unit that accepts a request, tracks the remaining time required to complete it, and reports its status back to the controller.

The workflow operates on a continuous loop representing discrete clock cycles. At the start of the simulation, the system initializes with a user defined number of servers. During each cycle, the system generates new incoming requests and pushes them into the primary queue. The LoadBalancer then checks for available servers and assigns the next request from the queue to any idle unit. Simultaneously, all active servers process their current jobs by decrementing the remaining time.

Dynamic scaling is handled at the end of each cycle. The LoadBalancer monitors the size of the request queue relative to the number of active servers. If the queue size exceeds eighty times the server count, a new server is provisioned to handle the load. Conversely, if the queue size drops below fifty times the server count, a server is deallocated to optimize resource usage. This cycle repeats for the specified duration, after which the system terminates and outputs final performance statistics.
